# PRD: 생활체육 체육시설 위치 확인 및 예약 서비스

---

## 1. 5W1H 기획 배경

| 구분 | 내용 |
| :--- | :--- |
| **Who (대상)** | 풋살, 테니스, 배드민턴 등 생활체육을 즐기는 동호인 및 개인 |
| **What (서비스)** | 주변 체육시설 위치 확인, 실시간 빈자리 조회 및 예약/결제 서비스 |
| **When (시기)** | 퇴근 후, 주말 등 체육 활동을 희망하는 시점 |
| **Where (장소)** | 사용자의 현재 위치 또는 검색된 특정 지역 기반 |
| **Why (목적)** | 시설 예약 과정의 파편화(전화, 개별 웹사이트) 해소 및 정보 접근성 향상 |
| **How (방법)** | 모바일 앱(LBS 기반)을 통해 시설 검색부터 결제까지 One-Stop 인터페이스 제공 |

## 2. 각 기능 상세 정의

**A. 위치 기반 시설 검색 (Location-based Search)**
* **기능 요약:** 지도 API를 활용한 반경 내 체육시설 핀(Pin) 표출 및 리스트 제공
* **데이터 필드:** 시설명, 종목, 위치(위도/경도), 평점, 현재 예약 가능 여부(Boolean)
* **제약 사항:** 초기 렌더링 시 사용자 반경 5km 이내 데이터만 Fetch하여 로딩 최적화 적용
* **연동 API:** Kakao Maps API 또는 Naver Maps API

**B. 시설 상세 정보 및 일정 조회 (Facility Details & Schedule)**
* **기능 요약:** 특정 시설 선택 시 상세 정보 및 날짜/시간별 예약 현황 제공
* **데이터 필드:** 시설 이미지, 이용 요금, 부대시설(샤워실, 주차장 등), 타임테이블(1시간 단위 Time block)
* **제약 사항:** 타임테이블 조회 시 실시간 동시성 제어 적용 (Redis 분산 락 활용 권장)

**C. 예약 및 결제 (Reservation & Payment)**
* **기능 요약:** 선택한 타임블록에 대한 결제 및 예약 확정
* **프로세스:** 타임블록 선택 -> 결제 수단 선택 -> PG사 연동 결제 -> 예약 완료 확정
* **연동 API:** 포트원(PortOne) 또는 토스페이먼츠(Toss Payments)

**D. 예약 내역 관리 (My Page)**
* **기능 요약:** 예약 내역 확인, 취소 및 환불 처리
* **제약 사항:** 시설별 환불 규정(이용 24시간 전 100% 환불 등)에 따른 자동 환불 금액 계산 로직 백엔드 필수 구현

## 3. UX Flow

1.  **앱 실행:** 위치 정보 권한(GPS) 승인
2.  **메인 화면 (지도):** 현재 위치 기반 주변 체육시설 마커 렌더링
3.  **검색 및 필터링:** 종목, 날짜, 시간대 조건 입력 및 필터 적용
4.  **상세 페이지 진입:** 시설 마커 탭하여 상세 정보, 요금, 리뷰 확인
5.  **일정 선택:** 예약 가능한 빈 타임블록 선택
6.  **결제 진행:** 예약자 정보 확인 및 외부 PG 연동 결제 완료
7.  **예약 완료:** 예약 번호 및 입장용 QR 코드 생성, 알림 발송

## 4. 기술 스택

| 분류 | 기술 스택 | 목적 및 채택 사유 |
| :--- | :--- | :--- |
| **Frontend** | Flutter | iOS/Android 크로스 플랫폼 단일 코드베이스로 빠른 MVP 구현 |
| **Backend** | Spring Boot | 안정적인 트랜잭션 처리 및 결제 비즈니스 로직 구현 |
| **Database** | PostgreSQL, PostGIS | RDBMS 구조 및 LBS(위치 기반) 공간 쿼리 효율적 처리 |
| **Cache** | Redis | 실시간 예약 동시성 제어 및 세션 데이터 관리 |
| **Infra** | AWS (EC2, RDS, S3) | 서버 호스팅, 데이터베이스 구축, 시설 이미지 등 정적 리소스 저장 |
| **3rd Party** | PortOne API | 신용카드, 간편결제 등 통합 결제 시스템 연동 |

## 5. MVP 로드맵

* **1~2주차 (기획 및 설계):** 화면 와이어프레임 확정, API 명세서 작성, DB 스키마 설계, 클라우드 인프라 세팅
* **3~4주차 (핵심 기능 개발):** 인증 로직, 지도 API 연동, 시설 데이터 구축 및 위치 기반 검색 API 구현
* **5~6주차 (예약 및 결제 개발):** 타임테이블 조회 API 구현, 예약 동시성 제어 로직 적용, PG사 연동 및 테스트 결제
* **7주차 (부가 기능 개발):** 예약 내역 관리, 취소/환불 로직 구현, 알림 서비스 연동
* **8주차 (테스트 및 배포):** QA 및 버그 픽스, 앱스토어/플레이스토어 심사 요청 및 정식 배포

## 6. 리스크 및 대응 전략

| 리스크 요소 | 문제점 | 대응 전략 (개발 관점) |
| :--- | :--- | :--- |
| **예약 동시성 문제** | 인기 시설 특정 시간에 다수 유저가 동시 결제 시도 시 중복 예약 발생 | Redis 기반 분산 락(Distributed Lock) 구현으로 결제 진입 시 타임블록 선점 처리 |
| **시설 예약 노쇼(No-Show)** | 예약 후 미방문으로 인한 시설 관리자의 영업 손실 | 100% 사전 결제 시스템 강제 적용 및 DB에 환불 위약금 테이블(Penalty Policy) 설계 |
| **오프라인 데이터 비동기화** | 전화 예약과 앱 예약 간의 데이터 불일치로 인한 오버부킹 | 시설 관리자용 어드민 페이지(Web) 제공 및 예약 블록 수동 잠금(Lock) API 구축 |
